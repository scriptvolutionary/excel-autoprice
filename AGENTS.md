# AGENTS.md

Документ фиксирует единые правила разработки для текущей и будущих итераций проекта.

## 1. Базовые принципы

- Архитектура: Clean Architecture (`domain`, `application`, `infrastructure`, `presentation`).
- Бизнес-логика не должна зависеть от UI, Excel-движка и конкретной БД.
- Любое изменение функционала должно проходить через тесты и логирование.

## 2. Кодировка, язык, локаль

- Все текстовые файлы в репозитории сохранять строго в `UTF-8` (без BOM).
- Кириллица разрешена и ожидается везде:
- комментарии в коде;
- сообщения об ошибках;
- пользовательские тексты GUI/CLI;
- лог-файлы, отчеты импорта, отчеты аллокации.
- Не использовать транслит, если текст предназначен пользователю.
- Для переносимости путей использовать `pathlib.Path`, не конкатенировать пути строками.

## 3. Правила по слоям

- `domain`:
- только модели, value objects, бизнес-правила и стратегии;
- без импорта `pandas`, `openpyxl`, `sqlite3`, `PySide6`.
- `application`:
- orchestration/use cases;
- зависимости только через интерфейсы и абстракции;
- стабильно поддерживаемые публичные контракты API.
- `infrastructure`:
- адаптеры БД/Excel/PDF;
- маппинг во внутренние модели;
- технические детали форматов.
- `presentation`:
- CLI/GUI;
- только сбор входных параметров и отображение результатов.

## 4. Комментарии и документация

- Комментарии писать только там, где логика не очевидна.
- Комментарии объясняют «почему», а не «что» делает строка.
- Для сложных методов использовать docstring с контрактом входа/выхода и ограничениями.

## 5. Работа с данными и безопасностью

- Ключ идентификации товара: только `SKU`.
- Любой импорт без SKU в обычном режиме — ошибка формата.
- Legacy-режим без SKU допускается только ограниченно и должен явно логироваться.
- Любая автоматическая коррекция количества по кратности должна помечаться флагом.
- Все нераспознанные строки при импорте попадать в отчет ошибок.

## 6. Миграции и SQLite

- Схема БД версионируется таблицей `schema_version`.
- Изменение структуры таблиц только через миграции.
- Миграции должны быть идемпотентны.

## 7. Тестирование

- Обязательные уровни:
- unit-тесты доменных правил;
- integration-тесты импорт/аллокация/экспорт;
- e2e-тесты ключевых пользовательских сценариев.
- Перед релизом обязательно проверять сценарии с кириллицей в путях и данных.

## 8. Качество и стиль

- Стиль: `ruff + mypy + pytest`.
- Явная типизация публичных интерфейсов обязательна.
- Не допускать скрытых fallback-решений без логирования.
- Сообщения об ошибках должны быть диагностируемыми и на русском.

## 9. Релиз и эксплуатация

- Для операторов поставляется `.exe` (PyInstaller), для разработки — Python-проект.
- В релизе должны быть:
- конфиг по умолчанию;
- каталог логов;
- пример входного шаблона;
- инструкция оператора.
